3. WHAT THE SITE CURRENTLY DOES

The public site:
- loads shared restaurant data from static JSON
- falls back to a compatibility mirror if the primary JSON path fails
- falls back again to an embedded snapshot in index.html if shared JSON cannot be loaded
- renders mapped restaurants on a Leaflet map
- shows a results list with score, subtitle, address, and actions
- supports search across restaurant names and subtitles
- supports score filtering
- supports sorting by highest score, lowest score, A to Z, and nearest
- supports browser geolocation and nearby-only filtering
- supports fit-to-visible-results behavior
- opens review videos in a new tab
- uses a `Get directions` action that prefers `googlePlaceUrl`, then `directionsUrl`, then a generated Google Maps fallback
- includes a mobile "Show full map" toggle

7. MANUAL INTAKE WORKFLOW

Current manual intake flow:
1. Open admin/add-review.html.
2. Enter the restaurant details.
3. Copy the generated JSON object.
4. Paste it into data/new-reviews.json.
5. Run node scripts/update_locations.js.
6. Run node scripts/refresh_static_artifacts.js.
7. Review the diff.
8. Commit and push.

What the admin helper does:
- collects the standard restaurant fields
- supports an optional Google Maps place/share URL field
- keeps directions URL support as the fallback field
- defaults city to Chicago and state to IL
- outputs JSON in pretty or one-line form
- does not write directly to the dataset

8. SHEET SYNC / AUTOMATION WORKFLOW

GitHub Actions workflow:
- .github/workflows/sync-sheet.yml

Current behavior:
- can be run manually with workflow_dispatch
- also runs hourly on a cron schedule
- reads GOOGLE_SHEET_CSV_URL from GitHub secrets
- downloads the published CSV
- accepts optional Google Maps place/share URL columns and maps them to `googlePlaceUrl`
- keeps only approved rows
- blocks obvious temporary/test entries
- sends approved rows through the same merge logic used by local intake
- writes data/locations.json and locations.json
- stages data/geocode-cache.json when enrichment cache changes
- commits and pushes data updates back to the repository

9. SCRIPT RESPONSIBILITIES

scripts/update_locations.js
- merges incoming restaurant rows into the canonical dataset
- reads data/new-reviews.json by default or an override file with --input
- applies manual fixes
- preserves stronger existing values when possible
- preserves a good existing `googlePlaceUrl` when incoming data leaves it blank
- writes both data/locations.json and locations.json
- clears data/new-reviews.json after merge unless --keep-new-reviews is used

scripts/sync_sheet.js
- downloads the published intake CSV
- canonicalizes supported sheet headers
- supports `googlePlaceUrl` header variants such as Google Maps URL, Google Place URL, Google Maps Place URL, Place URL, and Maps URL
- filters to approved rows
- blocks temporary/test data
- hands importable rows to scripts/update_locations.js

scripts/location_enrichment.js
- builds full addresses
- creates directions URLs only when `googlePlaceUrl` is absent and a usable `directionsUrl` is also missing
- geocodes missing coordinates through Nominatim when needed
- stores lookup results in data/geocode-cache.json

scripts/refresh_static_artifacts.js
- mirrors data/locations.json to locations.json
- rebuilds super_goode_locations.csv from the canonical dataset
- preserves the existing CSV header structure, adds `googlePlaceUrl` to the export schema, and keeps legacy columns
- refreshes the embedded fallback DATA array in index.html

12. CURRENT MAINTENANCE RULES

When updating the web map data:
- edit through the intake flow or approved sheet sync when possible
- treat data/locations.json as canonical
- keep locations.json mirrored to the source
- prefer `googlePlaceUrl` when a true Google Maps place/share link is available
- refresh super_goode_locations.csv from the canonical data
- do not remove CSV columns just because values are blank right now
- keep the index.html fallback snapshot aligned with the source dataset
- validate JSON files, CSV row count, and stale doc references before pushing

13. SUMMARY

The current Super Goode web map repo is a static GitHub Pages project backed by a 229-restaurant dataset. The repo now uses data/locations.json as the canonical source, keeps locations.json as a required mirror, supports an optional first-class `googlePlaceUrl` field, preserves a 21-column CSV export schema, and maintains an embedded fallback dataset inside index.html.
