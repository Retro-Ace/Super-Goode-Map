SUPER GOODE WEB MAP CURRENT BREAKDOWN

Last updated: 2026-03-27

1. PROJECT OVERVIEW

The Super Goode web map is a static restaurant discovery website built around Phil Goode's review data. It presents reviewed restaurants on an interactive map and in a browsable results list, with scores, subtitles, addresses, directions links, and review video links.

The project solves a simple but valuable problem: Phil's review content lives across social/video links, while the map site needs one clean place where people can browse all reviewed restaurants, sort them, search them, and quickly open directions or the review itself.

The live repo and site exist to keep that map experience fast, static, easy to host, and easy to update without a backend.

3. CURRENT WEB MAP FUNCTIONALITY

The website currently does the following:

- Loads restaurant data from a shared static JSON dataset
- Renders all mapped restaurants on a Leaflet map
- Shows a styled sidebar/panel with restaurant cards
- Lets users search by restaurant name and subtitle
- Lets users filter by score using a dropdown
- Lets users sort by highest score, lowest score, A to Z, or nearest to the user
- Lets users use current location and enable nearby filtering
- Lets users fit the currently visible map pins
- Shows score tiers with a small legend
- Opens map popups with title, subtitle, address, review link, and directions link
- Opens review links in a new tab/window
- Opens directions links to Google Maps
- Highlights selected restaurants/cards/pins

Map behavior:
- The map is built in index.html using Leaflet
- Pins are color-coded by score tier
- Selecting a restaurant can fly the map to that restaurant and open the popup
- The app syncs visible markers to the current filter/search result set
- The map includes a mobile "Show full map" toggle

Restaurant browsing behavior:
- Search matches both item.name and item.subtitle
- Search is case-insensitive
- Score filtering is currently done through the score dropdown only
- Nearby mode filters to places within roughly 10 miles when user location is active
- Results can be sorted by distance when the user has location enabled

Score display / tier behavior:
- Score tiers are:
  - 9.0 and up
  - 8.0 to 8.9
  - 7.0 to 7.9
- Current dataset count is 219 restaurants
- Current score distribution is:
  - 18 restaurants at 9.0+
  - 141 restaurants in the 8.x range
  - 60 restaurants in the 7.x range

Directions / review link behavior:
- Directions links resolve from directionsUrl when present
- If directionsUrl is missing, the app can fall back to a Google Maps directions/search URL
- Review links are normalized at runtime before opening, so query junk such as Instagram tracking parameters is stripped while preserving the core content path

Desktop and mobile experience at a high level:
- Desktop uses a two-pane layout: map on one side and the main browsing panel on the other
- Mobile uses a top map section, a dark controls/header area, and a separate independently scrolling results area below
- The mobile results area is intentionally scrollable on its own
- The mobile design has been tuned repeatedly to make the web map feel more app-like without changing the static hosting model

5. ADMIN / INTAKE WORKFLOW

There are currently two web-map-side ways to add restaurants.

Manual intake path:
- Open admin/add-review.html
- Enter restaurant info
- Generate a JSON object
- Paste that output into data/new-reviews.json
- Run node scripts/update_locations.js
- Review the console summary
- Commit and push

admin/add-review.html role:
- Single-file helper page
- No build tools required
- Generates valid JSON matching the dataset shape
- Supports compact one-line output and pretty output
- Built for quick manual additions

Sheet / form path:
- A Google Form or Google Sheet can be used as intake
- The sheet is published as CSV
- The workflow only publishes rows where approved is a yes/true-style value

Approval logic:
- Approval is handled in scripts/sync_sheet.js
- Only approved rows are passed into the merge flow
- Unapproved rows are ignored

6. SCRIPTS / AUTOMATION

scripts/update_locations.js

Purpose:
- Safely merges incoming restaurant rows into data/locations.json

What it does:
- Reads:
  - data/locations.json
  - data/new-reviews.json by default, or an override input file with --input
  - data/manual-fixes.json
- Normalizes names and text
- Prevents obvious duplicate merges
- Tries to merge into an existing row when names match cleanly
- Preserves stronger data already in the dataset
- Applies manual fixes last
- Writes both:
  - data/locations.json
  - locations.json
- Clears data/new-reviews.json after merge unless --keep-new-reviews is used
- Prints a summary of Added / Updated / Skipped / Ambiguous / Errors

Important merge behavior:
- It does not blindly overwrite stronger existing data
- It upgrades missing data when the incoming row adds something useful
- It blocks obvious temporary/test entries such as TEST BURGER and Test Pizza Place

scripts/sync_sheet.js

Purpose:
- Pulls approved intake rows from a published CSV URL and feeds them into the same merge pipeline

What it does:
- Reads GOOGLE_SHEET_CSV_URL from environment/secrets
- Downloads the published CSV
- Parses the CSV locally
- Canonicalizes expected fields
- Keeps only approved rows
- Blocks obvious temporary/test entries
- Writes a temporary JSON intake file
- Calls scripts/update_locations.js with that temp file as input

What is automated vs manual:
- Manual:
  - editing data/new-reviews.json
  - using admin/add-review.html
  - optionally running update_locations.js locally
- Automated:
  - scheduled CSV sync via GitHub Actions
  - committing refreshed data/locations.json and locations.json back to the repo when approved rows change the dataset

11. KNOWN CONSTRAINTS / CAVEATS

- This is still a single-file front-end app for the main UI, so most UI logic lives in index.html
- The root locations.json file is a mirror, which means changes should keep both JSON files in sync
- The automated sheet sync depends on correct external setup:
  - a published CSV URL
  - the GitHub secret GOOGLE_SHEET_CSV_URL
- If the external CSV is not published correctly or the secret is missing, sync automation will fail
- data/manual-fixes.json and data/new-reviews.json exist but are currently empty
- extraction_report.md is useful context, but it reflects the original extraction pass rather than the full current maintenance workflow
- node_modules is present locally, but it is not the important long-term source material for understanding the web map
