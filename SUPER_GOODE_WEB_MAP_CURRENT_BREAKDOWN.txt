3. WHAT THE SITE CURRENTLY DOES

The public site:
- loads shared restaurant data from static JSON
- falls back to locations.json if the primary JSON path fails
- falls back again to the embedded DATA snapshot in index.html if shared JSON cannot be loaded
- renders mapped restaurants on a Leaflet map
- shows a results list with score, subtitle, address, and actions
- opens review videos in a new tab
- uses `Get directions` with the current priority:
- googlePlaceUrl
- directionsUrl
- generated Google Maps fallback

Current search/filter behavior:
- searches across restaurant name, city, and subtitle or dish text
- uses normalized case-insensitive matching
- supports score filtering
- supports sorting by highest score, lowest score, A to Z, and nearest to the user
- preserves existing nearby/filter behavior on desktop

9. MANUAL INTAKE WORKFLOW

Current manual intake flow:
1. Open admin/add-review.html.
2. Enter the restaurant details.
3. Add googlePlaceUrl when a trustworthy place/share URL is available.
4. Keep directionsUrl as the fallback field when needed.
5. Copy the generated JSON object.
6. Paste it into data/new-reviews.json.
7. Run node scripts/update_locations.js.
8. Run node scripts/refresh_static_artifacts.js.
9. Review the diff, then commit and push.

Current admin helper behavior:
- collects the standard restaurant fields
- supports optional googlePlaceUrl
- supports optional directionsUrl
- defaults city to Chicago and state to IL
- outputs pretty or one-line JSON
- does not write directly into the dataset

10. SHEET SYNC / AUTOMATION WORKFLOW

Workflow:
- .github/workflows/sync-sheet.yml

Current behavior:
- can be run manually with workflow_dispatch
- also runs hourly on a cron schedule
- reads GOOGLE_SHEET_CSV_URL from GitHub secrets
- downloads the published CSV
- keeps only approved rows
- supports the Google Form `Request Type` field
- treats `Add location` rows as normal add/update intake
- treats `Remove location` rows as exact-location removals
- blocks obvious temporary or test add rows
- accepts optional Google Maps place/share URL columns and maps them to googlePlaceUrl
- passes approved rows through scripts/update_locations.js
- writes data/locations.json and locations.json
- refreshes super_goode_locations.csv and the embedded fallback DATA snapshot in index.html
- commits data updates back to the repository

Current remove safety rules:
- same-name restaurants at different addresses are treated as different locations
- remove requests only delete one exact matched location, not every location with the same name
- remove matching prefers review URL first, then exact name + address + city + state, then a very tight coordinate match
- ambiguous or weakly identified remove rows do not delete anything

Supported place-link header variants:
- Google Maps URL
- Google Place URL
- Google Maps Place URL
- Google Maps Place / Share URL
- Google Maps Share URL
- Place URL
- Place Share URL
- Maps URL

11. SCRIPT RESPONSIBILITIES

scripts/update_locations.js
- sanitizes incoming rows
- merges add rows into the canonical dataset
- removes rows only when a remove request matches one exact real location
- preserves stronger existing values when possible
- preserves a good existing googlePlaceUrl when incoming data leaves it blank
- keeps same-name different-address locations separate during merge
- applies manual fixes
- writes both data/locations.json and locations.json
- clears data/new-reviews.json unless --keep-new-reviews is used

scripts/sync_sheet.js
- downloads the published sheet CSV
- canonicalizes supported headers
- filters to approved rows
- normalizes Request Type into add vs remove behavior
- blocks obvious temporary/test add rows
- maps optional place/share URL columns into googlePlaceUrl
- hands importable rows to scripts/update_locations.js
- runs scripts/refresh_static_artifacts.js after successful sheet updates so CSV and embedded fallback stay aligned

scripts/location_enrichment.js
- builds full addresses
- auto-builds best-effort Google Maps place-style search URLs from restaurant name plus street-level address data when googlePlaceUrl is missing
- creates directionsUrl only when googlePlaceUrl is absent and a usable directionsUrl is also missing
- geocodes missing coordinates through Nominatim when needed
- stores lookup results in data/geocode-cache.json

scripts/refresh_static_artifacts.js
- mirrors data/locations.json to locations.json
- rebuilds super_goode_locations.csv from canonical data
- preserves the 21-column export schema
- refreshes the embedded DATA array in index.html

14. SUMMARY

The current Super Goode web map repo is a static GitHub Pages project backed by a 432-restaurant canonical dataset. The repo now uses data/locations.json as the source of truth, keeps locations.json as an exact mirror, exports a refreshed 21-column CSV, and maintains an embedded fallback snapshot in index.html. Search works across restaurant names, cities, and dish or subtitle text. Mobile uses a compact map-first layout with overlay Fit and Locate controls, a fullscreen map toggle, a white header bar, and a white scrolling results area.
